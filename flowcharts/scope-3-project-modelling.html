Foundation Phase · Requirement 02

Scope 3 Project Modelling — Foundational Update

Flowchart mapping the structural changes needed to extend the platform beyond Scope 2 energy modelling. This is the architecture layer — it defines the expanded data schema, multi-metric input logic, carbon-led value modelling, and integration with comparisons and strategies.

Why this matters: The current platform assumes all projects are energy-based (kWh), financially positive (ROI-driven), and Scope 2. This ticket removes those three assumptions and makes the modelling engine generic enough to handle any emission source, any unit, and any value logic — including projects that cost more but reduce carbon. Otto's "numbers come in, crunch a couple, numbers come out" principle governs the design.

Core change
Generic project input layer
Regardless of project type, every project feeds into the same structure: activity metric + unit + volume + emission factor + cost. The calculation engine doesn't care what the input type is — it applies the factor and produces a carbon and cost output. This is the "numbers in, crunch, numbers out" principle.

User selects activity metric + unit
Lookup
Platform queries emission factor library
Matches the activity metric and unit to the appropriate DEFRA/DESNZ factor. Library covers Scope 1, 2, and 3 categories. Factors stored with source, year, and unit mapping.
Decision

Yes — Standard factor
Auto-apply

kgCO₂e per unit applied automatically. Tagged as "DEFRA 2025" or equivalent source. User can see but doesn't need to enter it.
No — Custom needed
Manual
User enters custom factor
Free input for kgCO₂e per unit. Flagged as "Custom" in all outputs. Required for novel activity types or proprietary data. Org-level override option for admins.
Both paths continue
Conversion
Unit conversion framework
Where needed, the platform converts between units using built-in conversion tables (e.g. litres to kWh for fuels, tonnes to kg). This ensures consistent carbon calculations regardless of the input unit the user chose. All conversions are transparent and auditable.
Calculation
Carbon impact calculated
Baseline: baseline_volume × emission_factor = baseline tCO₂e
Projected: projected_volume × emission_factor = projected tCO₂e
Net impact: baseline − projected = carbon saving (or increase)

Same formula regardless of project type. The factor and unit do the work.
3. Carbon-Led Value Modelling
Carbon and cost outputs calculated
Decision
What is the value driver?
The platform must support three value models. The current platform only handles the first. This is the critical architectural shift — not every project saves money, but it still has value.
Financial ROI
Existing
Cost savings + carbon reduction
Traditional energy efficiency. Positive ROI, payback period, and carbon saving. Current model — no change needed. Display: savings, payback, IRR, carbon.
Carbon-led
New
Cost increase + carbon reduction
e.g. diesel to HVO: costs more, but 95% carbon reduction. ROI is negative but carbon value is high. Display: carbon saving, cost of carbon abatement (£/tCO₂e), compliance impact. Must not deter customers.
Blended
New
Mixed value — portfolio context
Some projects save money, others cost money but reduce carbon. When combined in a strategy, the portfolio can be net-positive financially while achieving carbon targets. This is the "commercially competitive net zero" logic.
Critical UX requirement: Projects with negative ROI must not be visually penalised or shown as failures. The current red/green profit framing would discourage customers from modelling legitimate carbon interventions. The UI needs a dual-axis presentation — financial impact AND carbon impact — where both are treated as first-class outcomes.
Output
Dual-value project summary
Every project outputs two parallel tracks:

Financial track: cost change (£), payback (if applicable), annual saving/cost
Carbon track: tCO₂e change, % reduction, abatement cost (£/tCO₂e), scope attribution

Both tracks feed into comparisons and strategies. Neither is subordinate to the other.
4. Integration — Comparisons & Strategies
Projects created (any type, any scope)
Comparisons
Multi-scope project comparison
Users can compare projects across scopes and types side by side. A diesel-to-HVO switch can be compared against an LED upgrade — normalised on carbon saving, cost, or abatement cost. The comparison engine uses the generic output format (financial + carbon tracks).
Strategies
Three-scope strategy modelling
Strategies can now pull in Scope 1, 2, and 3 projects. The strategy view aggregates financial and carbon impacts across all included projects — showing the portfolio-level business case. Cost-positive energy projects can offset cost-negative carbon interventions, demonstrating "commercially competitive net zero".
Reporting
Compliance & reporting alignment
Projects tagged by scope and fuel type automatically feed into compliance scheme reporting (SECR, ESOS). The expanded schema means all emission sources are captured at the point of modelling — no manual re-entry for compliance.
Scenario modelling
Future scenario extensions
The generic schema supports future scenario modelling — projected emission factors, future fuel costs, carbon pricing (ETS/CBAM), and what-if analysis. This ticket doesn't build those features, but the data structure must not block them.
Platform supports full net-zero pathway modelling
Scope 1 + 2 + 3 · Any fuel · Any metric · Financial + carbon value
