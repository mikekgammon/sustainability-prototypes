Alternative Fuel Modelling Integration

Flowchart mapping the fuel type data model, input logic, carbon factor application, fuel switching engine, and integration with compliance reporting and strategies. Builds on top of the Scope 3 Foundation schema.

Dependency: This feature requires the Scope 3 Foundation (Requirement 02) to be in place — specifically the expanded project data schema, multi-metric inputs, and carbon-led value modelling. The fuel type framework plugs into that generic structure.
Context: Many customers — manufacturing, data centres, fleet-heavy orgs — rely on fuels beyond electricity and gas. To fully support SECR and enable broader decarbonisation modelling, the platform needs to handle diesel, LPG, biomethane, HVO, kerosene, biomass, propane and more. This is distinct from the broader Scope 3 emissions accounting expansion — this ticket is specifically about fuel-based energy inputs and switching projects.

2. Consumption & Cost Input Logic
Fuel type selected, defaults loaded
Input
User enters consumption volume
Numeric input with the pre-selected unit. User can switch unit if needed (e.g. litres to tonnes for bulk fuel). Platform handles unit conversion behind the scenes using energy content values.
Decision
Unit matches default?
If the user changed the unit from the default, the platform needs to convert before applying the emission factor.
Yes — Direct apply
Straight-through
Volume × factor = tCO₂e
No conversion needed. Emission factor applied directly to the entered volume.
No — Convert first
Conversion
Convert via energy content
e.g. user enters tonnes of diesel, platform converts to litres using density, then applies per-litre emission factor. Conversion chain is logged and auditable.
Both paths continue
Cost input
User enters current and projected fuel costs
Two cost fields: current unit price (£ per litre/kg/kWh) and optional projected unit price for future cost modelling. Platform calculates annual cost from volume × unit price. Future: potential integration with fuel market pricing APIs, but manual entry always available.
Output
Fuel source record created
Record stored with: fuel type + volume + unit + emission factor + source + annual cost + tCO₂e + scope + compliance tags. This record feeds into the project summary, comparisons, and strategies via the Scope 3 Foundation schema.
3. Fuel Switching Engine
User initiates a fuel switch project
Baseline
Select current fuel (source)
User picks the fuel being replaced. Can select from existing fuel sources already on the project, or define a new one. The platform loads the current volume, unit cost, and emission factor as the baseline.
Replacement
Select replacement fuel (target)
User picks the new fuel. Platform loads its emission factor and default unit. User enters the replacement volume — may be 1:1 (e.g. diesel to HVO drop-in) or different (e.g. gas boiler to heat pump requires unit change).
Decision
Same unit type?
Is the replacement fuel measured in the same unit as the source fuel?
Yes — Direct comparison
e.g. diesel → HVO
Like-for-like volume
Both measured in litres. Direct comparison on carbon and cost per litre. Simple delta calculation.
No — Normalise first
e.g. gas → electric
Normalise via energy content
Convert both fuels to kWh equivalent using energy content values. Compare on a per-kWh basis for carbon and cost. Log the conversion path.
Both paths continue
Calculation
Fuel switch impact analysis
Platform calculates three deltas:

Carbon change: source tCO₂e − replacement tCO₂e = net saving
Cost change: source annual cost − replacement annual cost = net cost impact
Consumption change: volume difference (if applicable)

Outputs include: % carbon reduction, absolute tCO₂e saving, abatement cost (£/tCO₂e), and payback period (if cost-positive).
Decision
Cost impact direction?
Does the fuel switch save money or cost more? This determines the value framing (links to Scope 3 Foundation carbon-led value model).
Saves money
Win-win
Positive ROI + carbon saving
Display both financial and carbon benefits. Standard project framing. e.g. switching from expensive fuel to cheaper bio alternative.
Costs more
Carbon-led
Carbon saving at a cost premium
e.g. diesel → HVO. Show carbon saving as primary value. Display abatement cost (£/tCO₂e). Frame within portfolio strategy where other projects offset the cost.
Cost neutral
Pure carbon
No financial impact, carbon only
Rare but possible. Display carbon saving. No financial metrics needed beyond confirming zero cost impact.
4. Compliance & Reporting Alignment
Fuel sources and switches saved to project
Auto-tagging
Compliance schemes tagged from fuel type
Each fuel type in the library carries compliance scheme mappings. When a fuel source is added, the platform automatically tags which schemes it's reportable under. SECR requires all Scope 1 fuel emissions. ESOS requires energy use breakdowns. These tags are inherited by the project and propagated to strategies.
Integration
SECR reporting module
Alternative fuel data flows directly into SECR. No re-entry. Fuel type, volume, emission factor, and tCO₂e are all captured at modelling time. The SECR module (separate ticket, in progress) pulls from the same data. This ticket provides the data; SECR provides the output format.
Integration
Comparisons module
Fuel switching projects appear in project comparisons alongside energy efficiency projects. Compared on carbon saving, cost impact, abatement cost. The generic output format (financial + carbon tracks from Scope 3 Foundation) ensures comparability across project types.
Integration
Strategy module
Fuel switches can be included in multi-scope strategies. A strategy might combine LED upgrades (Scope 2, cost-saving) with a diesel-to-HVO fleet switch (Scope 1, cost-increasing) — the strategy view shows the net portfolio position. "Commercially competitive net zero" in practice.

5. Extensibility & Future Scope
Extensible framework
Beyond fuels — same data model applies
The fuel type framework is deliberately generic. The same structure (activity type + unit + volume + factor + cost) supports future emission sources without further architectural changes:
Future
Fleet fuel consumption
Same fuel library, but applied to vehicle fleet contexts. Distance-based and fuel-based metrics. Telematics API integration potential (e.g. Virta for EV).
Future
Refrigerant gases
kg-based inputs, high-GWP emission factors. Same structure — different factor library entry.
Future
Waste-related emissions
Tonnes to landfill, recycling rates. Activity-driven with waste-specific emission factors.
Future
Fuel market pricing API
Auto-populate current fuel costs from market data. Similar to electricity/gas market integrations. Manual entry always available as fallback.
