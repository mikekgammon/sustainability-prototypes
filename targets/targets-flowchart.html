Targets V2

How targets work in V2 — lifecycle, live tracking, and the relationship to Strategy V2.
personMike Gammon · Lead Product Designer calendar_todayApril 2026 descriptionCompanion to Strategy V2

Targets V2 is the rework of the Targets page, designed alongside Strategy V2 as the other half of the same user loop — set target → design strategy → track progress. The current Targets page is lightweight: a form to create a target, a static reduction path chart, no actuals overlay, no connection to the strategies meant to deliver it, and only Carbon reduction targets.

V2 closes those gaps and delivers against Otto's three Asana sub-tickets — multi-metric targets (Carbon, Energy, Cost), non-linear reduction pathways (checkpoints and curves, not just straight lines), and live progress tracking (M&T actuals overlaid on the target path with variance states). The Target ↔ Strategy relationship is also made explicit — which is the core logic decision this flowchart exists to settle.

Design principles

  • Multi-metric: Carbon · Energy · Cost
  • Trajectories can be non-linear
  • Same variance engine as Strategy V2
  • Live actuals, not static forecasts
  • Progress is the hero, not the form
  • Many-to-many with strategies
Current state · V1
The V1 Targets page presents as a basic list + form: users define an aim, pick benchmark year, set scope and reduction %, and see a static projected reduction path. It works, but only supports Carbon targets (no Energy or Cost), only linear reduction paths, no live tracking, no variance states, no strategy linkage, and the chart uses teal/purple (not brand meter colours). It also only supports Draft/Live — no terminal states for achieved or missed targets. Targets V2 addresses all of these while reusing the same input fields.

Target ↔ Strategy relationship

Key logic decision

The central question this flowchart exists to answer: how tightly should Targets and Strategies be bound in the UI? Four candidate models — tabs, separate pages deep-linked, target-centric, or strategy-centric — each implying a different data model and user flow.

Proposal

Separate pages, deep-linked, with many-to-many relationship

A target can be served by multiple strategies. A strategy can contribute to multiple targets. This is a many-to-many relationship, not a parent-child one — which rules out tabs (implies 1:1) and any owner-centric model (implies one-way hierarchy). Targets and Strategies stay as separate pages, each with its own purpose, linked by explicit references and a shared data layer (M&T, variance engine, emissions baseline). The user's journey moves between them via deep links.

Metric-matching rule

A strategy contributes to a target only if it produces that target's metric as an output.

Strategies produce all three metrics by default — carbon impact, energy consumption change, and cost impact — so in practice any well-modelled strategy can contribute to any target metric. The rule matters at the edges: a strategy modelled only for cost savings (with no carbon impact) cannot contribute to a Carbon target. The UI should make this explicit when linking strategies.

Targets page
The progress dashboard
"Am I on track?"
Shows every live target, its lifecycle state, aggregated actuals from linked strategies, and a deep link into the contributing strategies. Read-heavy. This is where a CSO / sustainability lead spends their time.
Strategies page
The design workspace
"What am I doing?"
Build, compare, and iterate strategies. Each strategy surfaces the targets it contributes to as metadata. Interaction-heavy. This is where an energy/sustainability manager works day-to-day.

Alternatives considered and why they fall short:

Alternative A
One page, tabs (Target | Strategy | Progress)
Implies a 1:1 relationship between a single target and a single strategy. Breaks down the moment a company has multiple targets (2030 Scope 1&2 plus 2038 Net Zero) served by overlapping strategies.
Rejected · doesn't model many-to-many
Alternative B
Target-centric — strategies nest inside
Targets "own" strategies. Works for simple cases but a strategy often contributes to several targets simultaneously — nesting forces duplication or arbitrary primary ownership.
Rejected · hierarchy is wrong shape
Alternative C
Strategy-centric — targets nest inside
Same problem inverted. A single target spanning multiple strategies can't be shown coherently if it lives under one strategy.
Rejected · hierarchy is wrong shape
Alternative D
Merged single page — Targets & Strategies combined
Collapses two distinct user intents ("am I on track" vs "what am I doing") onto one screen. High information density, poor navigability at scale, doesn't match the Schemes-quality benchmark.
Rejected · conflates two jobs

Dependencies

What Targets V2 rests on

Most dependencies overlap with Strategy V2 — by design, since the pair share a variance engine, a data layer, and most of the M&T plumbing. Strategy V2 is now itself an upstream dependency: the actuals Targets V2 displays are aggregated from linked strategies.

Dependency 1
Strategy V2
Targets V2 reads actuals from strategies that have contributed to them. Without Strategy V2's live tracking layer, there's no actual line to plot on the target path.
In flight · design review
Dependency 2
Shared variance engine
Target variance (actual vs target path) and Strategy variance (actual vs forecast) should run through one engine with one set of thresholds. Build once, used twice.
To align · Otto call
Dependency 3
M&T API
Indirect — Targets V2 consumes M&T data via linked strategies rather than directly. Still affected if coverage gaps exist at monthly granularity.
Partial · confirm coverage
Dependency 4
Scope 3 Foundational Update
If users can set Scope 3 targets before Scope 3 data quality is solid, the actuals side will be unreliable. Decision needed on whether to gate Scope 3 targets until the update ships.
Designed · gating TBC
Dependency 5
Emissions data store
Benchmark year selection relies on historical emissions being complete and reliable for the chosen period. Existing data model — no change expected.
In place
Dependency 6
Schemes UI pattern
Same design blueprint as Strategy V2. Monthly breakdown, stacked chart, M/Q/Y toggle, period controls — all inherited from the Schemes visual system.
Blueprint in place

Open logic questions

Resolve before wireframing

Eight questions to close before Targets V2 wireframes. Three of them overlap with the Strategy V2 open questions — worth resolving both in the same session with Otto to keep the pair consistent.

1
Strategy attribution — how does a strategy "contribute" to a target? Automatic via metric + scope overlap (any Scope 1 strategy with carbon outputs counts toward any Scope 1 Carbon target), or manual linking by the user? And how explicit should the metric-matching rule be in the UI?
Otto
2
Multi-metric targets — can a single target span metrics (e.g. "reduce carbon 75% AND save £2M by 2038"), or is each target strictly one metric? Simpler to keep them separate; but some sustainability reporting expects combined commitments.
Otto · Rochelle
3
Sub-targets / milestones — should a target support nested sub-targets (e.g. "50% by 2030" sitting inside "Net Zero by 2038")? Or are they treated as separate targets that happen to share a theme? SBTi-aligned reporting may push us toward explicit sub-targets.
Otto · Rochelle
4
Trajectory modes — how many do we offer? Linear is simplest. Milestones (user-placed checkpoints) reflects real projects. Curved (front- or back-loaded) models realistic decarbonisation. Ship all three from day one, or start with linear + milestones?
Otto
5
Achieved / Missed UX — when a target resolves to a terminal state, does it archive automatically, stay visible indefinitely, or prompt the user to create a successor target? CSRD audit trail implications either way.
Rochelle
6
Scope 3 gating — do we allow users to set Scope 3 Carbon targets now, before the Scope 3 Foundational Update is built? Actuals would be unreliable but users may still want to declare intent.
Otto
7
Variance thresholds — inherited from Strategy V2. Platform defaults (±5% / ±15%), customer-configurable, or per-target override? Answer should be the same for both Targets and Strategy.
Otto
8
Alert delivery — also inherited from Strategy V2. If a target drifts to At Risk / Off Track, do we notify externally (email / Teams) or surface only in-platform? Tie-in with Client Notifications.
Otto · Rochelle
