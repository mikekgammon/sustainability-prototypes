Dependency Detail

Scope 3 Foundation → Alternative Fuel Modelling Hard

Alternative Fuel Modelling requires the expanded project schema from Scope 3 — specifically multi-metric activity units (litres, tonnes, m³, not just kWh), the emission factor library (DEFRA/DESNZ lookups), and carbon-led value modelling (projects where cost increases but carbon decreases). Without these foundations, the fuel switching engine has no structure to plug into.

Scope 3 Foundation → Project Upload Template Indirect

The upload template's field definitions map to the project data model. If Scope 3 expands that model (adding scope, activity_metric, emission_factor_id, value_driver), the template needs to include those fields. Building the template before the schema is finalised means rework. The AI proposal scraping feature also relies on knowing which fields to extract into.

Universal Admin Access → All items Integration

No hard dependency in either direction — Admin Access operates at the identity and permissions layer, not the data model. However, it enables cross-account operations (duplicating projects between clients, bulk-uploading across accounts) that make the other three items more powerful in multi-tenant deployments. Can be built in parallel.

All Foundation items → Comparisons & Strategies Downstream

All four Foundation items feed into the Strategy V2 and Comparison modules downstream. Scope 3 + Alt Fuel populate multi-scope project data; Upload Template enables bulk creation; Admin Access enables cross-account strategy views. This is why the Foundation Phase matters — it unblocks the entire modelling and analysis layer above.
