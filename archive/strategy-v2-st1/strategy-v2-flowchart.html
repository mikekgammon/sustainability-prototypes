Strategy V2

How the page works — user journey, data flows, M&T integration, and scenario logic.
personMike Gammon · Lead Product Designer calendar_todayApril 2026 descriptionPrepared for Otto review

Strategy V2 is the rework of the Strategy page — currently the weakest screen in the platform (one red bar per year, no drill-down, no live data). This flowchart establishes the logic before the UI design to make sure the data architecture, calculation path, and retention loop are all sound.

The page sits on three foundations: the Project Calculation engine (builds the forecast), the M&T API (supplies the actual), and the Scenario modelling engine (compares alternatives). Variance detection on top of these creates the retention loop — the reason customers stay on the platform after deployment.

Design principles

  • Match Schemes for quality
  • Live data is the centre of gravity
  • Monthly base, aggregate up
  • Compare on shared axes only
  • Variance surfaces drift, doesn't hide it

Dependencies

What Strategy V2 rests on

Four upstream dependencies. One is ready, two are in-flight, one needs confirmation. Scope 3 Foundational Update status determines how much of the engine work gets redone vs reused.

Dependency 1
M&T API
Existing integration. Need to confirm field coverage is sufficient for the red/dashed line comparison at monthly granularity.
Partial · confirm coverage
Dependency 2
Project Calculation engine
Existing. The generic unit-agnostic principle (Otto) means no per-source rework. But the Improved Project Calc & Financial Modelling item in Phase 1 will alter outputs.
In flight · coordinate
Dependency 3
Scope 3 Foundational Update
Already designed (sent to Otto). Affects which intervention types can feed into strategies and how multi-scope stacking works.
Designed · awaiting build
Dependency 4
Advanced Scenario Modelling
Energy cost forecasting, long-term carbon modelling. The scenario comparison layer here should reuse this engine, not rebuild it.
Phase 2 · align scope
Dependency 5
Targets V2
Not a hard dependency, but Strategy V2 and Targets V2 share a natural user loop: set target → design strategy → track progress. Design them together.
Aligned · not blocking

Schemes UI pattern
The design blueprint. Month-by-month breakdown, stacked graphs, time-frame zoom — Schemes already does this well. Strategy V2 should borrow heavily.
Blueprint in place

Open logic questions

Resolve before wireframing

Six questions the flowchart can't answer alone. Answers shape the wireframes, so best to close these out with Otto before moving to UI.

1
Variance thresholds — who sets them? Platform defaults (±5% / ±15%), customer-configurable, or per-strategy override? Different answers imply different settings patterns in the UI.
Otto
2
Alert delivery — is the drift / breach state purely in-platform (red dot + banner), or does it trigger email / Teams notifications? Ties into the Client Notifications workstream.
Otto · Rochelle
3
Concurrent strategy limit — how many strategies can a user run in parallel per project? 3? 5? Unlimited? Performance and UI density both affected.
Otto · Dev
4
Targets V2 linkage — do we treat Strategy and Target as one connected journey (single page tabs) or as linked-but-separate pages? Otto's call on information architecture.
Otto
5
MVP scope for Otto review — is the first wireframe round covering the full vision (create + track + compare), or just the tracking view with existing strategies? Scoping affects next-sprint size.
Otto · Mike
6
Retired strategy handling — when a user promotes a new baseline, what happens to the old one? Archive, hide, or retain visible for audit trail? CSRD implications.
Rochelle
